# Idea Export PPT

## Purpose

Turn an idea into a concise, persuasive PowerPoint deck with a clear audience, message, narrative, proof, and call to action. Use this when the user needs a file for a meeting, executive discussion, workshop, email attachment, or offline review.

The deck may be general-purpose or person-specific. For person-specific decks, use public or provided context to make the message relevant to the stakeholder's role, likely priorities, and decision path.

The vibe is confident and generous: fewer slides that say more, less clutter, more decision energy.

## Workflow

1. Explore the idea before building.
2. Narrow the message into one audience, one desired decision, and one CTA.
3. Propose the deck storyline and slide count.
4. After the user confirms, create or modify the PowerPoint deck.
5. Render or otherwise verify the deck before delivery when tooling is available.
6. Provide the PPTX path, summary, and CTA.

## Discovery Questions

Ask only what is needed:

- Who is the audience?
- What meeting, decision, or discussion is this deck for?
- What should they understand, believe, decide, or do?
- What proof, examples, numbers, stories, or references support the idea?
- What objections or risks should the deck answer?
- What exact CTA or decision should appear at the end?
- How long should the deck be: 3, 5, 8, 10, or 12 slides?
- Should the tone be executive, visionary, technical, commercial, educational, or personal?

## Person-Specific Decks

Use this branch when the user provides a person, profile URL, LinkedIn URL, headshot, or asks to gear the deck to a specific stakeholder.

- Capture the target person, profile URL, image, relationship, idea, desired CTA, and tone.
- Use web research only when the user asks or provides a public URL.
- Mention only facts supported by public sources or the user's provided context.
- Do not infer private facts, personal attributes, or confidential strategy.
- Ask before using a public profile image unless it is clearly intended for public professional use.

## Deck Pattern

Use this structure unless the idea calls for a different one:

1. Title / decision ask.
2. Why now.
3. Current pain or missed opportunity.
4. The idea.
5. What changes.
6. Proof or evidence.
7. Plan and responsibilities.
8. Risks and safeguards.
9. Decision / CTA.

For very short decks, compress to:

1. Decision ask.
2. Context and problem.
3. Proposed idea.
4. Proof and risk handling.
5. Next step.

## PowerPoint Build Rules

- Build an actual `.pptx` deck, not a textual outline, when the user asks for a deliverable.
- Prefer clean, executive-readable slides over dense documents.
- Use one clear message per slide.
- Keep slide titles action-oriented.
- Use visuals when they clarify the idea: simple diagrams, process flows, screenshots, charts, or icons.
- Include speaker notes only when useful for presenting.
- If a template or brand deck is provided, follow it.
- If no template is provided, use restrained typography, consistent spacing, and a simple color system.
- Before sharing externally, confirm whether the content is confidential, internal-only, or safe to distribute.

## Verification

Before finalizing:

- Render or inspect the deck when tooling is available.
- Check that text fits on slides.
- Confirm images, charts, and diagrams display correctly.
- Confirm the title, decision ask, flow, CTA, and final slide are clear.
- Search for leftover names, old CTAs, previous image filenames, or template text.

## Output

When done, provide:

- The PPTX file path.
- A one-paragraph summary of the storyline.
- The exact CTA or decision slide.
- Any assumptions or unresolved questions.
