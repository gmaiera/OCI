# Oracle Design System

## Purpose

Help create Oracle-aligned, enterprise-ready experiences using Redwood-inspired principles: clarity, calm hierarchy, purposeful interaction, accessible structure, and business-user productivity.

This skill brings the polish pass: the moment where a useful artifact starts feeling trustworthy, reusable, and easy for a team to rally around.

Use this skill for:

- Oracle-style webpages and landing pages.
- Internal tools, dashboards, admin consoles, and data-heavy apps.
- PowerPoint or executive artifacts that should feel Oracle-aligned.
- UX reviews against enterprise usability and design-system expectations.
- Converting rough prototypes into coherent, polished business experiences.
- Organization-wide performance improvement where consistent UX, reusable patterns, and better decision artifacts matter.

## Source Guidance

Use official Oracle sources when specific design-system facts, component behavior, or implementation details matter. Prefer:

- Oracle Redwood public design material.
- Oracle JET Redwood theme and component documentation.
- Oracle Visual Builder, APEX, or product-specific Oracle UI documentation when the implementation target is named.

Do not invent exact Oracle brand rules, colors, spacing tokens, component APIs, or trademark guidance from memory. If exact compliance matters, verify against the official source or ask the user for the approved internal design reference.

Use `references/oracle-redwood-notes.md` only for lightweight orientation when a quick reusable checklist or artifact-pattern reminder is useful.

## Design Posture

Oracle-aligned design should feel:

- Clear rather than decorative.
- Productive rather than theatrical.
- Calm, structured, and executive-readable.
- Dense enough for professional work, but not visually crowded.
- Consistent across pages, slides, tables, forms, charts, and CTAs.
- Respectful of enterprise constraints: accessibility, security, auditability, localization, permissions, and scale.

Avoid:

- Consumer-app excess.
- Decorative gradients, blobs, bokeh, or purely atmospheric visuals.
- Oversized marketing compositions for operational tools.
- Card-heavy layouts where a table, form, or flow would be clearer.
- Fake Oracle branding or unsupported claims of compliance.

## Workflow

1. Identify the artifact type:
   - Website / landing page.
   - App / dashboard / operational tool.
   - PowerPoint / executive deck.
   - UX review.
   - Design-system rollout or enablement plan.
2. Identify the user and business job:
   - Executive decision.
   - Sales or stakeholder alignment.
   - Daily operations.
   - Data analysis.
   - Workflow completion.
   - Training, enablement, or adoption.
3. Choose the appropriate output pattern.
4. Apply Redwood-inspired hierarchy, spacing, tone, and interaction discipline.
5. Verify the result against usability, accessibility, and business clarity.
6. When useful, produce a reusable pattern, checklist, or template for organization-wide reuse.

## Output Patterns

### Website Or Landing Page

Use when the user wants a link or webpage:

- First viewport: product, idea, or decision signal immediately visible.
- Clear title, business value, and CTA.
- Structured sections with restrained visual rhythm.
- Use diagrams, screenshots, charts, or product visuals when they clarify the message.
- Keep CTAs practical: approve, reply, choose, book, comment, nominate, or start.
- Avoid a generic marketing hero if the purpose is internal alignment or executive decision.

### App Or Dashboard

Use when the user wants a working tool:

- Prioritize task completion, scanning, comparison, and repeated use.
- Use predictable navigation, filters, tables, forms, tabs, segmented controls, and clear empty/loading/error states.
- Use density carefully: show enough information for work, but maintain hierarchy.
- Make permissions, status, ownership, and next actions visible.
- Use charts only when they answer a decision question.
- Validate responsive behavior and text fit.

### PowerPoint Or Executive Artifact

Use when the user wants a deck:

- One message per slide.
- Decision-oriented slide titles.
- Calm hierarchy with consistent spacing.
- Simple diagrams and tables over decorative visuals.
- Final slide should make the decision, owner, date, or ask explicit.
- If an Oracle-approved template is supplied, follow it.

### UX Review

Use when reviewing an existing artifact:

- Start with findings that affect comprehension, trust, accessibility, or task completion.
- Identify inconsistent patterns and where reusable components would help.
- Separate "must fix" from "polish."
- Recommend a small set of reusable patterns rather than many one-off changes.

### Organization-Wide Design Improvement

Use when the goal is broader performance improvement:

- Define shared patterns for common work: briefings, dashboards, approvals, status pages, proposals, operating reviews.
- Create a reusable artifact library: page patterns, deck patterns, table patterns, CTA patterns, review checklists.
- Introduce lightweight governance: examples, do/don't rules, review gates, and champions.
- Track outcomes: faster decision cycles, fewer clarification meetings, better reuse, higher adoption, lower rework.

## Oracle-Aligned Checklist

Before finalizing, check:

- Is the audience and task obvious within seconds?
- Is the primary action clear?
- Does the artifact reduce cognitive load?
- Are hierarchy, spacing, labels, and states consistent?
- Are tables, charts, forms, and navigation built for repeated professional use?
- Are accessibility basics respected: contrast, labels, focus, keyboard behavior, alt text where relevant?
- Is the tone precise, plain, and enterprise-appropriate?
- Are any Oracle-specific claims or implementation details verified?
